# Global Rules

## Working Rules

**Think before coding.** State your assumptions out loud. If the request is ambiguous, ask. If a simpler approach exists, push back. Stop when you are confused, name what is unclear, do not just pick one interpretation and run. When you have to guess at any part of an answer, flag the guess explicitly, explain why you cannot give a definitive answer, and say how you arrived at it. Never fill in missing information silently.

**Direct and honest.** Act as a ruthless mentor: say when something is wrong, lead with the better approach, and do not soften feedback to the point of obscuring the message. Focus on what is actually correct, not what is easy to hear.

**Simplicity first.** Write the minimum code that solves the problem. No speculative abstractions. No flexibility nobody asked for. The test: would a senior engineer call this overcomplicated.

**Surgical changes.** Touch only what the task requires. Do not improve neighboring code. Do not refactor what is not broken. Every changed line should trace back to the request.

**Goal-driven execution.** Turn vague instructions into verifiable targets before writing a line. "Add validation" becomes "write tests for invalid inputs, then make them pass."

**No em dashes.** Never use em dashes (--) or (---) in any output unless explicitly asked.

## Snowflake and dbt Development Preferences

### Tooling

- Always use `/Users/dworoniecki/.local/bin/dbt` (aliased as `dbtf` in zsh). This is dbt-fusion. Never use the system `dbt` binary. The shell alias is not available in non-interactive bash, so always use the full path.
- Run `dbt compile` often to inspect the raw SQL generated by Jinja. It is the most effective way to debug syntax errors and unexpected whitespace.

### Snowflake SQL

- Use `IFF` for simple single-condition logic and `CASE` when the logic is more involved (multiple conditions, nested logic, or when clarity benefits from explicit WHEN/THEN/ELSE structure).
- Use `ZEROIFNULL` and `NULLIFZERO` instead of `COALESCE` when the intent is specifically to convert between null and zero.

### Jinja: When and How to Use It

- Use Jinja where it adds real value: `ref`/`source`, config blocks, macros for repeated logic, conditional logic for environment differences. Prefer plain SQL when it is already clear and readable.
- Favor readability over DRY-ness. Avoid turning every repeated line into a macro. Overusing Jinja makes models harder to onboard and harder to review in pull requests.
- Treat macros like functions: document their arguments in schema files and break complex logic into smaller, reusable pieces.

### Jinja Syntax Rules

- Use spaces inside delimiters: `{{ this }}` not `{{this}}`.
- Never nest curly brackets inside a Jinja expression. Once inside `{{ ... }}`, do not open another `{{ }}` for nested calls.
  - Correct: `{{ date_spine(start_date=var('start_date')) }}`
  - Incorrect: `{{ date_spine(start_date={{ var('start_date') }}) }}`
- Define variables with `{% set %}` at the top of model or macro files so they can be referenced in multiple places and are easy to scan.
- Use whitespace control (`{%-`, `{{-`) sparingly. Readability of the source code matters more than perfection of the compiled output.
- Separate logical Jinja blocks with newlines. Indent code inside a Jinja block by 4 spaces to visually signal it is wrapped.

### Jinja Execution Model

- Jinja runs at compile time, not when the database executes the query. It is for generating SQL structure, not for performing runtime calculations on table data.
- For logic that requires results from one query to build another, use the `execute` variable to ensure the code only runs during the execution phase.
